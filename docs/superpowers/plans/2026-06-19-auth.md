# Authentifizierung (Google-SSO + Allowlist) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Alle Web-Seiten und API-Routen hinter Google-SSO (Auth.js) schützen, Zugriff auf eine E-Mail-Allowlist beschränkt (fail-closed), durchgesetzt per Next.js-Middleware.

**Architecture:** Auth.js (NextAuth v5) mit Google-Provider, JWT-Session-Cookie. Ein `signIn`-Callback erzwingt die Allowlist. Eine Middleware schützt alle Routen: Seiten → Redirect zum Login, `/api/*` → 401. Die einzige rein testbare Logik (`isAllowedEmail`) ist unit-getestet; OAuth/Middleware werden per Build + aufgeschobener Live-Verifikation abgesichert.

**Tech Stack:** TypeScript · Next.js App Router · **next-auth@5 (Auth.js)** — neue Dependency · Vitest. Baut auf `main`.

## Global Constraints

- Library: Auth.js (NextAuth v5), Google-Provider, `session: { strategy: 'jwt' }` (keine DB-Sessions).
- Zugriff: E-Mail-Allowlist `AUTH_ALLOWED_EMAILS` (komma-separiert). **Fail-closed:** leere/fehlende Liste → niemand kommt rein.
- Durchsetzung: Next.js-Middleware vor allen Routen; Seiten → Redirect `/api/auth/signin`; `/api/*` (außer `/api/auth`) → `401 JSON`. Matcher schließt `api/auth`, `_next/static`, `_next/image`, `favicon.ico` aus.
- Secrets in `.env` (Bootstrap, NICHT im DB-Tresor): `AUTH_SECRET`, `AUTH_GOOGLE_ID`, `AUTH_GOOGLE_SECRET`, `AUTH_ALLOWED_EMAILS`, `AUTH_URL`.
- Auth.js v5 liest `AUTH_SECRET`/`AUTH_GOOGLE_ID`/`AUTH_GOOGLE_SECRET`/`AUTH_URL` automatisch aus der Env.
- Sync-CLI + CI laufen nicht durch die Middleware. Kein Schema-Change.

`src/lib/allowlist.ts`:
```ts
export function isAllowedEmail(email: string | null | undefined, raw: string | undefined): boolean {
  if (!email) return false;
  const allow = (raw ?? '')
    .split(',')
    .map((e) => e.trim().toLowerCase())
    .filter(Boolean);
  if (allow.length === 0) return false; // fail-closed
  return allow.includes(email.trim().toLowerCase());
}
```

import Google from 'next-auth/providers/google';
import { isAllowedEmail } from '@/lib/allowlist';

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [Google],
  session: { strategy: 'jwt' },
  callbacks: {
    signIn({ user, profile }) {
      const email = profile?.email ?? user?.email;
      return isAllowedEmail(email, process.env.AUTH_ALLOWED_EMAILS);
    },
  },
});

- [ ] **Step 1: Middleware implementieren**

`src/middleware.ts`:
```ts
import { auth } from '@/auth';
import { NextResponse } from 'next/server';

export default auth((req) => {
  if (req.auth) return NextResponse.next();

  const { pathname, href, origin } = req.nextUrl;
  if (pathname.startsWith('/api/')) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
  const signInUrl = new URL('/api/auth/signin', origin);
  signInUrl.searchParams.set('callbackUrl', href);
  return NextResponse.redirect(signInUrl);
});

export const config = {
  matcher: ['/((?!api/auth|_next/static|_next/image|favicon.ico).*)'],
};
```
*(Falls der Edge-Build den Google-Provider in der Middleware ablehnt: das Base-Config-Objekt — `providers`/`callbacks`/`session` — in `src/auth.config.ts` auslagern, `src/auth.ts` macht `NextAuth(authConfig)`, und die Middleware nutzt `NextAuth(authConfig).auth`. Mit reinem Google-Provider sollte der direkte Import jedoch bauen.)*
